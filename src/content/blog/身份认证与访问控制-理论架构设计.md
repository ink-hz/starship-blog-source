### 2.3 Opaque Token架构

#### Opaque Token验证流程

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant Gateway as API网关
    participant IAM as IAM系统
    participant Cache as Token缓存<br/>(Redis)
    participant Service as 后端服务

    Note over Client,Service: 阶段1: 令牌颁发
    Client->>IAM: 1. 认证请求<br/>(username + password)
    IAM->>IAM: 2. 验证凭证
    IAM->>Cache: 3. 存储令牌元数据<br/>token_abc → {user_id, roles, exp}
    IAM->>Client: 4. 返回Opaque Token

    Note over Client,Service: 阶段2: 令牌使用
    Client->>Gateway: 5. API请求 + Token
    Gateway->>IAM: 6. Token Introspection<br/>(验证令牌)
    IAM->>Cache: 7. 查询令牌信息
    Cache->>IAM: 8. 返回用户信息
    IAM->>Gateway: 9. 返回验证结果<br/>{active: true, user_id, roles}
    Gateway->>Service: 10. 转发请求<br/>附加用户上下文
    Service->>Gateway: 11. 响应
    Gateway->>Client: 12. 返回结果

    Note over Client,Service: 阶段3: 令牌撤销
    Client->>IAM: 13. 登出请求
    IAM->>Cache: 14. 删除令牌
    Cache-->>IAM: 令牌已失效
```

#### 令牌方案对比表

| 维度 | JWT | Opaque Token | Session ID |
|------|-----|-------------|-----------|
| **存储位置** | 客户端（自包含） | 服务端（引用） | 服务端 |
| **验证方式** | 本地验证签名 | 调用IAM Introspection | 查询Session存储 |
| **状态** | 无状态 | 有状态 | 有状态 |
| **撤销能力** | 困难（需黑名单） | 容易（删除即可） | 容易 |
| **性能** | 高（无需查询） | 中（需网络调用） | 中（需查询存储） |
| **可扩展性** | 优（无状态） | 中（依赖IAM） | 差（状态共享） |
| **安全性** | 中（信息可解码） | 高（无信息泄露） | 高 |
| **跨域支持** | 优 | 优 | 差（Cookie限制） |
| **适用场景** | 微服务、API | 金融、高安全 | 单体应用 |

## 网关认证鉴权架构

### 3.1 完整认证鉴权流程

```mermaid
sequenceDiagram
    participant Client as 客户端
    participant Gateway as API网关
    participant IAM as IAM系统
    participant Blacklist as 黑名单存储
    participant AuthZ as 授权服务
    participant Service as 后端服务

    Note over Client,Service: 阶段1: Token提取与格式验证
    Client->>Gateway: HTTP请求<br/>Authorization: Bearer <token>
    Gateway->>Gateway: 提取Token
    alt Token格式错误
        Gateway->>Client: 401 Unauthorized
    end

    Note over Client,Service: 阶段2: 签名与有效期验证
    Gateway->>IAM: 获取公钥(JWKS)
    IAM->>Gateway: 返回公钥
    Gateway->>Gateway: 验证签名
    alt 签名验证失败
        Gateway->>Client: 401 Invalid Signature
    end

    Gateway->>Gateway: 检查过期时间(exp)
    alt Token已过期
        Gateway->>Client: 401 Token Expired
    end

    Note over Client,Service: 阶段3: 黑名单检查
    Gateway->>Blacklist: 查询jti是否在黑名单
    Blacklist->>Gateway: 黑名单状态
    alt 在黑名单中
        Gateway->>Client: 401 Token Revoked
    end

    Note over Client,Service: 阶段4: 身份还原
    Gateway->>Gateway: 解析JWT Payload
    Gateway->>Gateway: 提取:<br/>• user_id (sub)<br/>• roles<br/>• permissions<br/>• tenant_id<br/>• agent_type

    Note over Client,Service: 阶段5: 授权决策
    Gateway->>Gateway: 路径匹配<br/>请求方法识别

    alt 简单RBAC(网关本地)
        Gateway->>Gateway: 检查roles
    else 复杂策略(调用授权服务)
        Gateway->>AuthZ: 授权请求<br/>{user, resource, action}
        AuthZ->>Gateway: Allow/Deny
    end

    alt 权限不足
        Gateway->>Client: 403 Forbidden
    end

    Note over Client,Service: 阶段6: 请求转发
    Gateway->>Service: 转发请求 + 用户上下文<br/>X-User-ID: user_123<br/>X-User-Roles: admin<br/>X-Tenant-ID: org_789

    Service->>Gateway: 业务响应
    Gateway->>Client: 返回结果

    Note over Client,Service: 服务信任网关,无需再次认证
```

#### 用户上下文传递方式对比

| 传递方式 | 优势 | 劣势 | 适用场景 |
|---------|------|------|---------|
| **HTTP Header** | 简单直接<br/>无需解析<br/>后端易获取 | Header可被篡改<br/>需内部网络信任 | HTTP/REST API<br/>内网服务 |
| **内部JWT** | 防篡改<br/>标准化<br/>可验证 | 需解析Token<br/>性能开销 | 跨服务调用<br/>需验证场景 |
| **gRPC Metadata** | 协议原生支持<br/>类型安全 | 仅限gRPC | 微服务通信 |
| **Service Mesh** | 自动注入<br/>透明传递 | 依赖Mesh基础设施 | Kubernetes环境 |

### 7.4 微服务架构中的身份传递

```mermaid
sequenceDiagram

    participant ServiceA as 服务A
    participant ServiceB as 服务B

    Note over User,ServiceC: 阶段2: 网关转换
    EdgeGW->>APIGW: 内部Token (短期JWT)<br/>HS256签名, 5分钟有效
    APIGW->>APIGW: 验证内部Token<br/>提取用户上下文

    Note over User,ServiceC: 阶段3: 服务间调用 - 方案A (Header传递)
    APIGW->>ServiceA: 请求 + Headers:<br/>X-User-ID<br/>X-User-Roles<br/>X-Tenant-ID

    Note over ServiceA,ServiceC: 服务A调用服务B
    ServiceA->>ServiceB: 透传Headers<br/>(原样传递用户上下文)
    ServiceB->>ServiceB: 从Header提取用户信息<br/>进行业务逻辑

    Note over User,ServiceC: 阶段4: 服务间调用 - 方案B (Service Mesh)
    ServiceB->>ServiceC: gRPC请求

    Note over ServiceB,ServiceC: Service Mesh自动注入<br/>用户上下文到Metadata
    ServiceC->>ServiceC: 从gRPC Metadata提取<br/>用户信息

    ServiceC->>ServiceB: 响应
    ServiceB->>ServiceA: 响应
    ServiceA->>APIGW: 响应
    APIGW->>EdgeGW: 响应
    EdgeGW->>User: 最终响应

    Note over User,ServiceC: 全链路保持用户身份<br/>每层可进行权限检查
```

## 总结与最佳实践

### 核心设计原则

| 原则 | 说明 | 实施建议 |
|------|------|---------|
| **最小权限** | 只授予完成任务所需的最小权限 | 默认拒绝，显式授权 |
| **职责分离** | 认证、授权、审计分离 | 独立的IAM服务 |
| **深度防御** | 多层安全控制 | 边缘+网关+服务多层验证 |
| **零信任** | 永不信任，始终验证 | 持续验证，短期凭证 |
| **可审计** | 所有操作可追溯 | 完整日志，不可篡改 |

### 方案选型矩阵

| 场景 | 用户规模 | 推荐方案 | 理由 |
|------|---------|---------|------|
| **初创公司** | < 1000 | Auth0/玉符 | 快速集成，免费额度 |
| **成长期** | 1000-10000 | Keycloak自建 | 成本控制，功能完整 |
| **大型企业** | 10000+ | Azure AD + Keycloak | 混合架构，各取所长 |
| **云原生** | 任意 | 云厂商IAM + Keycloak | 云资源用云IAM，应用用Keycloak |
| **多云环境** | 任意 | 统一IDaaS (Okta) | 跨云统一身份 |
